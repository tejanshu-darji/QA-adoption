8. Core Team Rules

Onboarding requirements and working standards

This section defines the onboarding requirements, daily working standards, and usage practices for team members using Claude across Simform engineering projects.

RuleExpected Practice
Use Claude in the right workspaceUse Claude Code in VS Code when project privacy and project-specific context must be preserved.
Isolate projectsCreate a separate local folder for each project so context does not carry across projects.
Stay within scopeWork only within your assigned project folder and do not modify files from unrelated projects.
Use project-scoped MCP configurationUse local MCP configuration for project-specific tools and avoid global connector setup when it could expose cross-project data.
Keep MCP personalMaintain a personal MCP configuration file within your own project folder and do not share it.

13. Do's and Don't

Operational habits that keep sessions focused
DoDon't
  • Use plan mode for any non-trivial task before asking Claude to make changes.
  • Keep one task per prompt so the session stays focused and the output stays accurate.
  • Watch context usage and start a fresh session or compact the conversation when the session gets too large.
  • Use clear prompts with file names, expected behavior, and acceptance criteria whenever possible.
  • Give Claude a way to verify its work, such as tests, expected outputs, or validation steps.
  • Use subagents for investigation or exploration tasks so the main session stays clean.
  • Keep your project instructions concise and relevant so important guidance does not get buried.
  • Enable only the MCP tools or connectors that are needed for the current project.
  • Save important outputs, decisions, and chats inside the project folder for future reference.
  • Review the suggested changes before applying them to confirm they match the intended scope.
  • Do not keep unrelated work in the same session for too long.
  • Do not ask Claude to handle multiple unrelated tasks in a single prompt.
  • Do not switch models in the middle of an active session unless there is a strong reason to restart the work.
  • Do not add or enable new MCP connectors mid-session unless it is necessary and approved.
  • Do not create an oversized project instruction file with information Claude can already infer from codebase.
  • Do not paste large raw logs or long command outputs into the conversation when a filtered summary will do.
  • Do not ask Claude to broadly explore the repository without scope when you only need a specific answer.
  • Do not accept changes without checking the active folder, current context, and affected files.
  • Do not use shared or global project configuration when local project isolation is required.
  • Do not modify files outside your assigned folder or project scope.

9. Workspace Initial Setup & Environment Configuration

VS Code, Claude Code, and Claude Desktop setup

VS Code & Claude Code Setup

  • Install Claude Code and the required VS Code extension.
  • Sign in and complete the required account setup, including GitHub authentication if your workflow requires it.
  • Create or open your assigned local project folder before starting any Claude session.
  • Create your personal MCP configuration within that folder if the project uses MCP tools.
  • Confirm that you are working only inside the correct project folder before running prompts or accepting changes.
  • Store important conversations, outputs, and notes in the same project folder so project history remains together.
  • If required: If a project does not use a dedicated project folder, follow a clear conversation naming format such as User - ProjectName.
  • If required: If MCP tools are needed, add only the local or project-scoped connectors required for that work. Do not enable broad access unless approved.

Claude Desktop Setup & Context Layers

  • Download and install Claude Desktop from claude.ai/download, then sign in.
  • Create a dedicated folder for AI-related project documentation so Claude Desktop has a clear, structured context source.

Claude Desktop works with three explicit layers of context that should be managed proactively:

  • Layer 1: Conversation Context: Everything discussed in the current chat session is retained as active context (e.g., investigating structural components).
  • Layer 2: Attached Files: Files uploaded directly (architecture blueprints, trace-logs, or schemas) become immediately inspectable.
  • Layer 3: Project Knowledge: Long-lived project documentation intentionally curated by the team within the workspace directory.

6. Best Practices For Access Types

How to use each Claude interface well

Claude Web

  • Upload complete requirements.
  • Ask for risks, gaps and edge cases together.
  • Use separate chats for separate features.
  • Use Projects for long-running initiatives.

Claude Desktop

  • Upload complete logs.
  • Include screenshots and videos.
  • Share expected vs actual behavior.
  • Provide environment information.

Claude Projects

  • Create separate projects per product or module.
  • Store testing standards and requirements.
  • Reuse project knowledge across releases.
  • Maintain dedicated regression projects.

Claude Code / CLI

  • Analyze repository architecture before making changes.
  • Use repository-wide searches before refactoring.
  • Generate reusable automation components instead of one-off scripts.
  • Validate generated code using existing framework standards.
  • Use CLI for multi-file updates and framework migrations.
  • Leverage terminal access for CI/CD investigations.
  • Use prompts that specify coding standards and project conventions.
  • Review generated pull requests before merging.

VS Code Extension

  • Use for focused tasks.
  • Keep context limited to relevant files.
  • Switch to Claude Code for larger changes.

Claude Co-Work

  • Use for team discussions.
  • Capture decisions and actions.
  • Review release risks collaboratively.
  • Use during requirement grooming sessions.

MCP

  • Use personal credentials.
  • Connect only required systems.
  • Leverage live project information.
  • Use for impact analysis.

7. QA Life-Cycle Workflows

Hand off tasks across interfaces from requirements to release

Maximize your sprint velocity by strategically handing off tasks across interfaces as development moves from requirements to release.

Workflow A: New Feature Validation & Automation Lifecycle

Phase 1: Requirement Analysis (Claude Web)
Paste a new feature specification into Web. Ask Claude to identify edge cases, data boundaries, and missing validation loops. Document your findings.
Phase 2: Test Orchestration (Claude Projects)
Move the validated features into your dedicated Release Project. Provide Claude with the historical manual test suites to generate updated Gherkin/Feature files mapping directly to your core standards.
Phase 3: Scripts & Page Objects (Claude Code / CLI)
Open the terminal within your framework repository. Direct Claude Code to automatically scan existing architecture, map page components, and build out complete, runnable automation scripts.
Phase 4: Sync & Polish (MCP)
Leverage MCP Jira connectors directly from your interface to map the newly generated automated scenarios back to the originating story id, keeping statuses aligned.

Workflow B: CI/CD Pipeline Defect Mitigation Lifecycle

Phase 1: Log & Environment Analysis (Claude Desktop)
A nightly build fails in staging. Open Claude Desktop, grant file system access or upload raw trace-logs along with browser failure screenshots. Have Claude diagnose the structural failure.
Phase 2: Repository Root-Cause & Refactor (Claude Code / CLI)
With the error isolated, hand off the context to Claude Code in your local CLI. Instruct it to seek out the flaky wait state or damaged DOM selector and refactor files globally across the repository.
Phase 3: Run Validation and Commit (Claude Code / CLI)
Instruct Claude Code to execute the targeted suite locally in the terminal. Once tests pass consistently, prompt Claude to stage, commit, and spin up a pull request directly from the shell environment.

Workflow C: Mid-Sprint Scope Modification & Regression Alignment

Phase 1: Scope Evaluation (Claude Web + MCP)
A critical change request arrives mid-sprint. Feed the altered requirement delta into Claude Web and use the Jira MCP tool to see which components are touched. Evaluate immediate delivery risk.
Phase 2: Automated Sweep (Claude Code / CLI)
Identify structural framework changes needed to cover the scope modifications. Direct Claude Code via terminal to locate affected custom variables and apply bulk modifications to parameter blocks concurrently.
Phase 3: Team Risk Alignment (Claude Co-Work)
Initiate an interactive session in Co-Work with the broader team. Review the updated coverage matrix, regression boundaries, and potential release side-effects collectively before kicking off signed runs.

10. First-Day Quick Start

Practical checklist and common pitfalls
StepAction Item Checklist
1Install Claude Code and open it in VS Code or Claude Desktop.
2Sign in and complete required account setups, including GitHub device handshakes.
3Create or open your assigned local isolated project folder.
4Run initial project setup configurations and verify active runtime scoping inside that folder.
5Create your personal local settings.json or .claude assets if required by the project.
6Audit and check active MCP tools; immediately disable unneeded cross-project connectors.
7Initiate an explicit small task, invoke plan mode to align target directories, and save history.

Common Failures & Remediation Paths

  • Unrelated tasks in one long session: Noise increases context pollution. Fix: Instantiate fresh operational chat threads or compact state properties regularly.
  • Vague requirements prompts: Causes architectural wandering. Fix: Provide target files, schemas, assertion rules, and verification checklists on line 1.
  • Skipping plan mode on major transformations: Induces loose code additions. Fix: Enforce structural alignment planning tasks prior to script execution.
  • Pasting massive raw terminal outputs: Blinds attention windows. Fix: Apply local log grep filters or extract concise traceback frames prior to ingestion.

The QA Golden Rules

  1. Keep each project strictly isolated in its own local folder.
  2. Use local MCP configuration and avoid unnecessary global access pathways.
  3. Plan first for non-trivial work and keep exactly one task target per prompt run.
  4. Keep context focused by starting fresh when tasks shift.
  5. Keep CLAUDE.md concise, clear, and focused solely on active framework properties.
  6. Review suggested structural transformations carefully before accepting workspace edits.

12. Security, Governance & Compliance Mandates

Strict operational restrictions for sensitive data
Strict Operational Restriction: Under no conditions are team members permitted to transmit sensitive, proprietary, or un-scrubbed parameters to public cloud models.

Prohibited Structural Elements:

  • Production API access tokens, keys, and authorization hooks.
  • Live customer database connectivity matrices and decryption secrets.
  • Personally Identifiable Information (PII) belonging to users or stakeholders.
  • Raw operational certificates, security assets, or encryption secrets.

Corporate Enforcement: All platform engineers and automation professionals remain wholly accountable for verifying and testing generated code structures prior to pushing to master build lines.
